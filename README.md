## Github 사용 규칙

### 커밋 메시지 규칙
- [INIT] : 초기 셋팅 시
- [ADD] : 새로운 기능 생성 시
- [UPDATE] : 만들어 놓은 기능의 변경점이 있을 시
- [FIX] : 기능의 버그가 있어 수정했을 시
- [REFACTOR] : 코드 리팩토링 시
- [DOCS] : 문서 수정 시
- [STYLE] : 코드 스타일 변경 시(코드 포맷팅, 코드의 기능 변경 없을 경우) 
- [DEL] : 삭제 시(중요 파일 삭제 시)
- `Summary`는 최대한 간결하게 작성한다.
  - ex) [ADD] Player 이동 로직 생성
  - 상세한 내용이 필요한 경우 `Description`에 자세히 작성한다.
  - 코드 변경 시에는 변경된 코드를 굳이 커밋 메시지에 포함하지 않는다.

### 커밋 메시지 규칙 (예외 상황)
- 유니티를 저장하지 않은 상태에서 커밋하여 재커밋이 필요한 경우
  - [ADD] Re: 이전 커밋 메시지 그대로 기입
  - [UPDATE] Re: 이전 커밋 메시지 그대로 기입
- 중요 파일의 이름 변경 및 폴더 이동 시에도 [UPDATE]를 사용한다.
  - ex) [UPDATE] 스크립트 정리를 위한 파일 이동
    - 파일 이동 시점을 추적하고, 프로젝트 변경 이력을 관리하기 위함이다.

### 커밋 수행 규칙
- 하나의 기능 단위로 커밋한다.
  - 작업 도중 백업용 커밋은 예외적으로 가능
- 하루 작업을 마무리할 때 커밋을 권장한다.
  - 커밋하기 애매한 상황이라면 하지 않는 것이 맞다.

### 브랜치 생성 규칙
- 브랜치 병합 후에는 반드시 삭제
- 브랜치 네이밍은 '기능명_작업자 이니셜' 형식을 기본으로 설정

### Pull Request 작성 규칙
- 담당은 송근형 명예시민이 맡는다.
  - 단, 담당자가 부재 시 대통령님이 진행한다.
  - 대통령님도 부재할 경우, 나머지 인원이 협의하여 병합을 진행한다.
- Pull Request를 올리고 반드시 팀 내에 공지한다.
- 담당자가 코드 확인후 Merge를 진행한다.
- 병합 과정에서 문제가 발생할 경우, 담당자를 포함한 3인 이상이 합의하여 진행한다.
  - 근무 시간 이후에는 `네트워크 프로젝트 팀` 스레드를 이용해 의견을 수렴한다.

### Merge 규칙
- [Merge] `Summary` 느낌으로 간략하되 명확하게 구현내용을 기입
